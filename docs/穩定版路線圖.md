# HostWitness 穩定版路線圖

本文件將「穩定版成熟度強化」拆成可執行的階段、交付物與驗收條件，供後續排程與 release gate 使用。

對應待辦來源：`docs\還有什麼要做的事項.md` §1。Phase 0 的邊界定義見 [穩定版邊界定義](D:/cursor/IR_script/HostWitness/docs/穩定版邊界定義.md)。

---

## 1. 目標

- 將 HostWitness 從「功能完整且可用」提升為「現場可辯護、可重現、可支援」的穩定版 DFIR 工具。
- 優先強化採集可說明性、輸出一致性、驗證流程與操作收斂。
- 後續功能擴充以不破壞穩定版邊界為前提。

## 2. 非目標

- 不以新增大量 parser / artifact 類型為本階段主要目標。
- 不優先投入 UI 美化、雲端同步、自動更新等與現場鑑識價值弱相關的項目。
- 不在未定義輸出格式相容政策前大改 `snapshot`、`manifest`、`SQLite schema`。
- 本輪 stable update 先以 UI / Core 本機流程為主；Agent 強化、遠端部署與獨立採集流程驗證延後至未來更新項目。

---

## 3. 推進順序

### Phase 0. 穩定版邊界定義

**目的**

- 先凍結「什麼算穩定版」，避免後續 hardening 目標漂移。

**交付物**

- 支援矩陣初稿：Windows 版本、權限需求、檔案系統、是否依賴 VSS。
- 模式定義：`Forensic Strict`、`Triage Fast` 的預期行為、限制與告警。
- 功能分級：`forensic`、`non-forensic`、`advanced` 三類邊界。
- 輸出物清單：`snapshot`、`manifest`、`SQLite`、log、raw artifacts。

**驗收**

- 有一份文件能回答「這版支援什麼、不支援什麼、哪些功能會影響現場」。
- UI、README、LIMITATIONS 的描述不互相矛盾。

### Phase 1. 可辯護性

**目的**

- 讓每次採集都能交代「做了什麼、沒做什麼、哪裡不完整」。

**交付物**

- 模式與告警落盤：
  - 記錄執行模式、權限、實際採集路徑、fallback、截斷、live/non-forensic 告警。
- `preflight check`：
  - 管理員權限、輸出路徑、可用空間、時間/時區、工具版本、VSS 狀態。
- `collection summary`：
  - 成功、失敗、跳過、截斷、證據位置、雜湊、告警清單。

**驗收**

- 任一輸出 bundle 都能回推出採集模式與限制。
- MFT / Registry / Snapshot 這類高風險路徑若降級或截斷，摘要中必定可見。
- 現場分析員不需要翻 UI log 才知道結果是否完整。

### Phase 2. 可重現性

**目的**

- 確保不同時間、不同機器、不同維護者對同一版本有一致結果。

**交付物**

- `snapshot` / `manifest` / `SQLite schema` 版本欄位與相容政策。
- golden sample 集合：
  - NTFS 1024/4096 record size
  - VSS 成功/失敗
  - Admin/非 Admin
  - ETW 節流
  - 輸出截斷
- 驗證矩陣：
  - Windows 10 / 11 / Server
  - Admin / non-admin
  - VSS available / unavailable
  - NTFS / non-NTFS
  - 磁碟忙碌 / EDR 存在 / 空間不足

**驗收**

- 每次 release 前有固定矩陣可跑。
- 舊版輸出至少能被新版辨識版本並給出明確處理結果。
- schema 或輸出格式異動時，有明確 migration 或 breakage 聲明。

### Phase 3. 操作一致性

**目的**

- 降低分析員自由拼裝流程造成的結果差異。

**交付物**

- 現場採集 profile：
  - `5 分鐘 triage`
  - `15 分鐘 standard`
  - `60 分鐘 full`
- SOP / runbook：
  - 何時用哪個 profile
  - 何時允許 high-risk 功能
  - 何時改用 offline / VSS / raw 路徑
- 高風險功能隔離：
  - dump、Live Registry、其他可能改變現場狀態的動作，需顯著標示並寫入摘要。

**驗收**

- 團隊內兩位分析員用同一 profile 對同一情境操作，輸出差異可控。
- 非鑑識功能不會混進「默認穩定版流程」。

**Phase 3 交付狀態（2026-03-19）**

- 操作員 profile 與 SOP：見 `docs/StableOperatorProfileAndSOP.md`（Forensic Strict / Triage Fast / 預設穩定流程、SOP 摘要表、UI 動作分級表）。
- 高風險隔離：Create Dump 選單已加 ToolTip 標示 high-risk、非預設鑑識流程；Registry Search 維持既有 Live 時 non-forensic 標示。

### Phase 4. Release Gate

**目的**

- 將「穩定版」變成可被重複套用的出貨流程。

**交付物**

- release checklist
- 驗證紀錄模板
- known limitations 更新流程
- 回歸測試與文件同步 gate

**驗收**

- 沒通過 checklist、驗證矩陣、文件同步，不出 stable release。

---

## 4. 建議先做的三件事

1. 模式與告警落盤
2. `preflight check` + `collection summary`
3. 正式驗證矩陣

理由：

- 這三項直接決定現場可辯護性與後續維護成本。
- 在這三項沒做完前，增加新功能的效益通常低於增加不確定性。

---

## 5. 每階段建議輸出

| 階段 | 建議輸出 |
|------|----------|
| Phase 0 | `docs/穩定版路線圖.md`、支援矩陣草案、模式定義 |
| Phase 1 | manifest/meta 擴充、preflight 報告、collection summary 範本（先以 UI / Core 為主） |
| Phase 2 | schema version、golden samples、驗證矩陣、migration 規則 |
| Phase 3 | profile 設定、SOP、UI 標示與摘要欄位 |
| Phase 4 | release checklist、驗證紀錄、stable release criteria |

## 6. Release Gate 建議

穩定版至少應滿足：

- build / test 全綠
- 驗證矩陣跑完且結果留檔
- 輸出格式版本明確
- 已知限制文件同步
- 高風險功能標示一致
- `preflight` 與 `collection summary` 可用
